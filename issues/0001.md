## Description

Set up the foundational Go project structure for the `issued` binary. This is Phase A from `RemoteSync.md` and is the prerequisite for all subsequent phases.

Deliverables:

- Go module (`cmd/issued/main.go`) with packages `auth`, `api`, `ws`, `admin`, `storage`, `db`, `wire`
- Postgres migration runner and initial schema (all tables from `RemoteSync.md § Postgres schema`)
- S3 client wiring; IAM role on EC2 (no static credentials)
- `GET /healthz` returning `200` with build SHA
- Structured JSON logging to stdout via `slog`
- `systemd` unit file (`issued.service`): `Restart=on-failure`, `User=issued`, listens on `127.0.0.1:8080`
- Apache 2 VirtualHost config: `mod_proxy` + `mod_proxy_http` for HTTP, `mod_proxy_wstunnel` for `/v1/events`; `ProxyTimeout 3600`; TLS via existing certbot setup
- Environment variable loading (`BASE_URL`, `DATABASE_URL`, `S3_BUCKET`, `AWS_REGION`, etc.)
- Single static binary build, ~15 MB

## Notes

**S3 client**: use AWS SDK v2 (`github.com/aws/aws-sdk-go-v2/service/s3`). Configure the client from env at startup — `S3_ENDPOINT` + `S3_PATH_STYLE` cover all S3-compatible providers; omitting static credential vars falls back to IAM role auto-detection on AWS. The `s3/manager` sub-package handles multipart uploads (Phase J).

Apache 2 is already running on the EC2 box with certbot managing TLS. The `mod_proxy_wstunnel` `ProxyPass` directive for `/v1/events` must appear before the catch-all `ProxyPass /` in the VirtualHost block or Apache will treat WebSocket upgrades as plain HTTP.

Admin HTML/CSS/JS should be embedded via `embed.FS` so the binary is fully self-contained.
