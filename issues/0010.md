## Description

Production hardening and UX polish across the full stack. This is Phase J from `RemoteSync.md`. Should not be started until Phases A–I are verified working.

Backend:

- Rate limits on auth endpoints (challenge, assertion, invite redemption)
- Request size caps (suggest 25 MB per file; larger needs multipart — design separately)
- Structured error responses with machine-readable codes
- Refresh token rotation: rotate on every use, revoke prior token, 30-second grace window for racing requests
- `events` table GC: configurable retention, default 24 h
- Audit log retention: 90 days hot in Postgres, daily roll-up exported to S3 indefinitely
- `/metrics` Prometheus endpoint (optional, add if monitoring is wired up)
- Nightly `pg_dump` → S3 (`lifecycle: keep 30 daily, 12 monthly`)

Admin console:

- "Add device" QR flow: signed-in user → "Add device" → QR encodes a short-lived registration token → scanned on new device → PassKey registered on new device

Mac client:

- Per-project subscription persistence across launches (restore `lastSeenEventId` on startup)
- "Remove local copy" button in Settings: leaves files in place on disk, removes the server binding (per `RemoteSync.md § Open questions #7`)
- Role-change handling: `403` on `PUT` after demotion → surface "your role changed" banner, park edit as `.local-conflict-*` file

## Notes

The `viewer` role is useful for stakeholders; `editor` is the common developer case; global `admin` is operator-only. Role demotions that arrive mid-session should not silently drop in-flight edits.
